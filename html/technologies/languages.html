Languages
Assembly Language
My first experience of microprocessors and digital electronics was at university during my engineering degree. My final year project involved working on an Intel 8048 processor. (ref. Intel MCS-48). The development environment was Daisy DNIX (ref. Daisy DNIX). This gave me an introduction to modular design techniques and embedded software debugging.

My next influential experience was when I began working with Telecom NZ. (ref. Telecommmunications) I worked for four years on a very large scale 32-bit assembly language NEAX-61E (ref. NEAX-61E) telephone exchange platform. The software was modular and highly complex. My role was to implement features now taken for granted such as CENTREX (ref. CENTREX), three-way calling, call waiting, call diversion, network handover tones.

We used a proprietary NEC operating system with task control, inter-task messaging, priority queuinmg and the concepts of context switching. Being a safety critical system meant strict adherence to the SDLC beginning with thoroughly reviewed and documented designs, implementations and unit tests. These went on to multiple forms of testing such as 'Soak', 'Stability', 'Integration' and finally 'Customer Acceptance'.

I was to take this experience and apply it in my next role working for Tait Electronics on the Motorola processors for their conventional and trunked radios. (ref. Motorola 68HC11)

Embedded C
I first began using Embedded C under an in-house mentorship program at Tait Electronics. As part of the new product development team I was trained in the intracacies of the language with a particular focus on Object Oriented Analysis.

Since this time I have only ever used C as my primary language. I have employed the OOA and OOD techniques I learned during this development in all of my subsequent projects and have found it to be extremely useful particularly with respect to portability, maintainability and comprehensibility. All this came at a small sacrifice of code space and execution time which by now was becoming less of an issue as processors became faster and more powerful. I have been able to debug easily and communicate designs clearly via the source code alone (though not without appropriate documentation).

C# MVVM
My introduction to C# has been relatively recent. I am very much a novice. However, I was given the opportunity whilst working for an automotive testing house, to manage a small team of Windows developers implementing a GUI for communicating with CANBUS decoders. I was able to teach myself the basics of the language by piggy-backing off their code and implementing basic interpreter code at the low level board interface. This was my first exposure to the concept of object oriented programming languages.

Following this I have used C# to allow me to provide simulated graphical interfaces to other products via serial comms allowing me to interact with prototype hardware in it's early stages of development. However, it is very much a 'tool' for me which I like to call upon when necessary during my normal embedded development.
